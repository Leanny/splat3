1. Introduction

Welcome to the fascinating world of Tableturf Battle! Whether you're a seasoned player or just stepping into this realm, leveling up is a quest we all embark upon. In this guide, I'll unveil some of the most efficient strategies to supercharge your leveling journey. A quick note: while I delve deep into the tactics, a basic understanding of the game's core concepts will come in handy.

2. Strategies

This section will focus on the main part of the article: How to level up fast and/or with barely any effort. Here we have two main categories: Singleplayer (SP) and Multiplayer (MP) strategies. SP strategies can be executed offline and alone. They are not really efficient, but reliable and easy to automate as well. The MP strategies involve two switches that play against each other.

2.1. Evaluating a Strategy

The metric I use to describe how good a strategy is, is "Points per Second". which means the more points you gain in a second, the faster you will level up. For simplicity sake I will assume that one level requires you to get 3333 EXP and that you start at level 50. Here are some key numbers:

Ideally we want to get as high as possible, but something like 1.2 EXP per Second is already enough to get to level 999 within a month.

2.2. Code Format

In order to describe certain aspects that can be coded a pseudocode language similar to python is used. Further the following assumptions are met:

  • player1 refers to the controller of player 1.
  • player2 refers to the controller of player 2.
  • current_frame refers to the current frame returned by the capture device.
  • controller.input(buttons, delay) refers to a method that inputs a list of buttons with a delay between each of press delayseconds.

2.3. Singleplayer

2.3.1. Auto Pass

This is the simplest strategy. It takes quite some time to level up, but it is really reliable and has no additional costs. For this strategy you go into a battle and simply pass every turn until you lose. Every loss will give you 40 EXP and takes roughly 2 minutes, which means we get roughly 0.33 EXP per Second. This task can easily be automated, i.e. with a Bluetooth Controller. The macro here to use is 

def skip_turn():
       player1.input([DPAD_UP, A, A, B], 0.02)
       sleep(1)

Repeating this sequence an infinite amount of time will let you pass every match and start a new match.

2.3.2. Clone Jelly

This method requires you to be on level 47 or higher as this unlocks the NPC opponent "Clone Jelly". This enemy has the specific trait that it copies your deck. With the correct strategy you can get up to 0.83 Points per Second when playing against a Level 1 Clone Jelly, 0.96 Points per Second when playing against a Level 2 Clone Jelly, or 1.08 Points per Second when playing against a Level 3 Clone Jelly. There are two main strategies to beat Clone Jelly reliably:

  1. Low Tile Count + 3-12 Finish: The deck here is pretty simple: it consists of your 14 lowest tile count cards and one card with a tile count of 12 and a special cost of 3. Your strategy is to build 3 special points and use the 3-12 card on the last turn to overlap the tiles of the enemy.
  2. Low Tile Count: This deck consists of 3 low tile count cards and 12 cards that have a higher tile count than the 3. As of Splatoon 3 Version 4.x the smallest possible deck that fulfils this criteria consists of 1 card of size 1, 2 cards of size 2, 5 cards of size 3, and 7 cards of size 4. Alternative you can also use 12 cards of size 5. The strategy is to place every card except for the low tile count ones, no specials needed. Clone Jelly Level 3 will always force a Draw out of this, against Clone Jelly Level 2 you will have a win rate of around 66% and against Clone Jelly Level 1 you will have a win rate of around 75% with this strategy.

2.4. Multiplayer

This section talks about the most efficient way of leveling. This requires additional hardware:

  • Two switch consoles
  • A capture card
  • Two controlling devices
  • Two copies of the game

I will not talk about the different possible controlling devices in detail, but my personal suggestion is either using an emulated Bluetooth Controller, as this technically allows you to control two switches with only one dongle, or using two arduino boards running a modified version of the Switch Fightstick PoC for optimal controller connection and speed. My personal recommendation is the Smart Program Controller by Brianuuu, which also includes a manual with all hardware and setup information you need.

2.4.1. Idea

The core idea for this farming method is that in a match against another real player, you get as many EXP as you get points in the round. This unfortunately caps at 100 points per player, but you still get the points no matter if you win, or lose. You also get the points when prematurely ending the match, which allows us to reduce the time of a match to less than 90 seconds while getting a lot of EXP.

2.4.2. Execution

The most simple way to gain a lot of EXP is to use the Auto Pass strategy on one switch, and play for as many points as possible. The deck to use here is your 15 cards with the biggest tile count. This method gave me on average 1.3 EXP per Second. Further, it is optimal to give up the match when you have 90 or more points, as the additional time to place another card is not efficient for the time it takes to select and place the card.

This method is simple and does not require much additional equipment besides a second switch and a second game. Theoretically this method can also be executed with a friend online, with the only problem being that you automatically disconnect after 4h in a lobby (see Splatoon 3 Patch Notes, Version 4.0.0 for them acknowledging the bug).

2.4.3. Automating Everything

Ideally we want to automate this whole process for both switches. In this section I will give an overview of various steps that are required for automating the process.

First I will define some functions that will be used in the code:

def confirm(controller):
       controller.input([A], 0.02)
   
   def cancel(controller):
       controller.input([B], 0.02)
   
   def skip_turn(controller):
       controller.input([DPAD_UP, A, A, B], 0.02)
       sleep(1)

Next I want to describe a state machine that describes a game:

stateDiagram-v2

       state "Get Frame" as gf

       Turn --> Turn: if points < 90

The advantages of a system like this is that it does not depend on the previous state and can basically be started from any point of the game. Further, a new state to handle a disconnect can also be introduced easily. I will now talk about some of the states in detail.

The Stage Selection is quite easy. We can either play on the Stage Main Street or on the Stage Double Gemini for the most reliable way to get points. Double Gemini is the stage with the most tiles, so even by placing cards randomly on the board we can assure a lot of points per game. On the other hand is Main Street the first map in the selection screen, which makes the Stage Selection part trivial. The code for this state could look like this: 

def handle_stage_selection(frame):
       cursor = get_current_cursor_position(frame)
       if cursor != target:
           move_cursor(cursor, target)
       else:
           confirm(player1)

This implementation ensures that all movement input comes through before confirming it.

The next state is the Deck Selection. We can use the same code base as for Stage Selection here, but my suggestion is that we assume that the deck we want to use is on the first slot for both players: 

def handle_deck_selection(frame):
       confirm(player1)
       confirm(player2)

For the Redraw state we have two choices: Either parse the current hand and redraw if we only have "low cards" or just accept it and confirm as fast as possible.

The Select Card state is a lot more interesting. In this state we pick a card that we want to place on the board. The AI algorithm can be a lot of things, depending on your map. In general it is advised to only look at unrotated cards, as rotations also rotate the start point of the card which makes it harder to correctly calculate the sequence for the movement. Further, it should always pick the card with the most tiles. For Main Street I suggest to play the card as close to your start point as possible, while on Double Gemini nearly everything will work. 

def handle_select_card(frame):
       skip_turn(player2)
       hand = get_hand(frame)
       field = get_field(frame)
       cursor = get_current_cursor_position(frame)
       turn = generate_turn(hand, field)
       if cursor != turn.hand:
           move_cursor(cursor, turn.hand)
       else:
           confirm(player1)
           return turn

For the Place Card state the difference between current position and target position is calculated and then the card is moved. This could look like this: 

def handle_place_card(frame, turn):
       current = get_current_card_position(frame)
       diff_x, diff_y = current - turn.target
       sequence = []
       if diff_x > 0:
           sequence.extend([DPAD_LEFT] * diff_x)
       elif diff_x < 0:
           sequence.extend([DPAD_RIGHT] * -diff_x)
       if diff_y > 0:
           sequence.extend([DPAD_DOWN] * diff_y)
       elif diff_y < 0:
           sequence.extend([DPAD_UP] * -diff_y)
       if len(sequence) == 0:
           # no moves to do
           confirm(player1)
       else:
           # move to position
           player1.input(sequence, 0.03)

After placing the card we repeat the process until we detect that our points are greater than 90 OR when no turn can be executed anymore. In that case we enter the state Give Up: 

def handle_give_up(frame):
       cursor = get_current_cursor_position(frame)
       if cursor == -1:
           # no give up screen up yet
           player1.input([PLUS], 0.02)
       if cursor == 0:
           # not giving up
           player1.input([DPAD_RIGHT], 0.02)
       if cursor == 1:
           confirm(player1)

After that we can either introduce more states or start a new round and be in state Wait: 

def handle_wait(frame):
       confirm(player1)
       confirm(player2)

2.4.4. Improving Automation

With a bot implemented as above we can expect roughly 0.8-0.9 EXP per Second. From my research I was able to identify the biggest bottlenecks: The safety nets that ensure that we can execute the bot from any given state. With a few assumptions we can improve the speed by a lot, but also make the bot more prone to errors. First, let's take a look at the new state diagram:

By removing every state except for Select Card, Wait and Give Up, we assume that both players have the deck in position 1, and that we always play on Main Street. Further, depending on the controller type we use, we might end up in an endless-loop as commands might not get through. But if none of that is a problem, then we can change the Select Card method to the following: 

def handle_select_card(frame):
       # always skip player 2
       skip_turn(player2)
   
       # we get the current hand and field
       hand = get_hand(frame)
       field = get_field(frame)
   
       # cursor is always 0 in our assumption
       cursor = 0
   
       # we assume that we have no rotations
       turn = generate_turn(hand, field)
   
       if cursor != turn.hand:
           # move to position
           move_cursor(cursor, turn.hand)
       # assume that we are correct with out placement
       confirm(player1)
       
       # we calculate the starting position based on the shape of the card
       current = get_current_card_position(turn.card)
   
       diff_x, diff_y = current - turn.target
       sequence = []
       if diff_x > 0:
           sequence.extend([DPAD_LEFT] * diff_x)
       elif diff_x < 0:
           sequence.extend([DPAD_RIGHT] * -diff_x)
       if diff_y > 0:
           sequence.extend([DPAD_DOWN] * diff_y)
       elif diff_y < 0:
           sequence.extend([DPAD_UP] * -diff_y)
   
       # press A at the end of the sequence    
       sequence.append(A)
       player1.input(sequence, 0.03)

The start position of a card can easily be calculated. For Main Stage we use the following assumptions:

  • The field size is 9×26 9\times26 > 9 × 26 .
  • Top left is (0, 0) (0, 0) (0,0).
  • The opponent start position is (4,3)(4, 3)(4,3).
  • Your start position is (4,22)(4, 22)(4,22).
  • Our card is shrunken, i.e. there is no full row or column with unset tiles only.
  • The position of a card is given by the coordinates of its top left tile.
  • hhh is defined as the height of the shrunken card, and www as the width.

With this optimization I was able to get to 1.4 Points per Second.

2.4.5. Reliable Movement

One of the big issues with the current approach is that we need to have long times between each DPAD input in order for it to register properly. In order to ensure that our next input is registered, we need to send a different output between these two, i.e. instead of sending [DPAD_UP, DPAD_UP] we should send something like [DPAD_UP, NOTHING, DPAD_UP] or even something like [DPAD_UP, R, DPAD_UP] works. But this is only a slight improvement. The trick that doubles the moving speed is by replacing every 2nd DPAD command with the left stick. So for example, instead of sending [DPAD_UP, DPAD_UP, DPAD_UP, DPAD_UP] we send [DPAD_UP, LSTICK_UP, DPAD_UP, LSTICK_UP]. This has the same properties as sending a different button and we can send another direction button even faster, allowing us to move as unhuman speed.
